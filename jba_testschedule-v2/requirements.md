## 1. プロジェクト概要

### 1-1. 背景

JBA社内のプロジェクト進行管理を、スプレッドシート＋Zoho手動管理から、Claude Codeで構築するWebダッシュボード（HTML+CSS+JS）に置き換える。

井尻さんがClaude Codeでわずか5分でプロトタイプ（ガントチャート＋稼働管理ダッシュボード）を作成。社長が「破壊的イノベーション」「Zohoより圧倒的に良い」と評価し、実装を即決定。

### 1-2. ユーザーと課題

| ユーザー | 拠点 | 役割 | 主な課題 |
|---------|------|------|---------|
| 瀬山さん | 大阪 | 進行管理主担当 | 全案件をスプレッドシートで一人管理（属人化）。ピーク時は毎朝口頭で進捗共有 |
| 役員 | — | 管理職 | タスクの所在（ボール）が不透明。品質チェック関所の形骸化 |
| 社長 | — | 意思決定者 | Zohoの手入力負担。全体進捗が水面下に潜る |
| 井尻さん | — | エンジニア・AI活用 | プロトタイプ作成者。実働4時間で基盤実装可能と見積もり |
| 原口さん | 東京 | ディレクション/制作進行 | 全体統括ディレクター不在。分業制で全体像が見えない |

### 1-3. 課題一覧（議事録より10件抽出）

| ID | 課題 | 影響度 | 対応する機能要件 |
|----|------|--------|----------------|
| P-01 | 進行管理が瀬山さんに属人化 | 高 | FR-01, FR-08 |
| P-02 | スケジュール引き直しが全て手作業 | 高 | FR-06 |
| P-03 | タスクの所在（ボール）が不透明 | **最高** | FR-02 |
| P-04 | 制作メンバーの稼働量が見えない | 高 | FR-03 |
| P-05 | 品質チェック関所が機能していない | 高 | FR-04 |
| P-06 | 全体進捗が見えない（水面下に潜る） | 高 | FR-05 |
| P-07 | 東京に全体統括ディレクター不在 | 高 | FR-01, FR-02 |
| P-08 | 人によってプロセス・書き方がバラバラ | 中 | FR-06 |
| P-09 | 台割りからフォルダを作る手作業が煩雑 | 中 | FR-07 |
| P-10 | リマインド・アラートが口頭/紙ベース | 中 | FR-05 |

### 1-4. 技術方針

- **形式**: 単一HTMLファイルSPA（既存プロトタイプ踏襲）
- **技術**: HTML5 / CSS3 / Vanilla JavaScript（フレームワーク不使用）
- **永続化**: localStorage（既存実装を踏襲）
- **外部依存**: なし
- **運用コスト**: 無料（NFR-02）
- **出力先**: `output/スケジュール/index.html`

## 2. 機能要件

### 2-1. 基本要件（議事録ベース FR-01〜08）

#### FR-01: ガントチャート / プロジェクト一覧

| 項目 | 内容 |
|------|------|
| **目的** | 全案件を横並びで俯瞰し、ページ/記事単位の進捗を把握 |
| **既存実装** | ✅ 基本実装済み（予定バー青/実績バー緑赤黄、今日線、日付範囲自動計算） |
| **改修内容** | ガントチャートに担当者ラベルを追加。横スクロール改善。50案件規模対応 |
| **xlsx知見** | 月別カレンダー×ページ/記事別のマトリクス構造が必須。全3ファイル共通のパターン |
| **実装優先度** | 最優先 |

#### FR-02: 担当者（ボール）明示

| 項目 | 内容 |
|------|------|
| **目的** | 各工程で今誰がボールを持っているかを一目で表示。宙に浮いたタスクを検知 |
| **既存実装** | ❌ 未実装（タスクに担当者フィールドなし） |
| **新規内容** | Taskに`assignee`フィールド追加。担当者マスター管理。ボール受け渡し履歴追跡。担当者フィルタリング |
| **xlsx知見** | JBA側（企画/取材/執筆/編集/デザイン）とクライアント側の2系統で担当者管理。ヘッダ行で担当者明示 |
| **実装優先度** | **最優先**（役員・社長が最も強調した課題） |

#### FR-03: リソース（稼働）ヒートマップ

| 項目 | 内容 |
|------|------|
| **目的** | メンバーごとのタスク量・想定時間を可視化し、パンクを未然防止 |
| **既存実装** | △ 部分実装（日別棒グラフあるが、担当者別の分解なし） |
| **改修内容** | 担当者×日付マトリクスのヒートマップ化。全プロジェクト横断集計。8h超セルを赤警告 |
| **xlsx知見** | 瀬山さん管理の大阪案件は50件規模。10社展開で特定メンバーがパンクするリスク |
| **実装優先度** | 高優先 |

#### FR-04: 品質関所（チェックゲート）

| 項目 | 内容 |
|------|------|
| **目的** | 各工程のチェックをシステム上で必須とし、チェック漏れ・品質低下を防止 |
| **既存実装** | ❌ 未実装 |
| **新規内容** | FMTの各工程にチェックリスト定義。関所到達時に全項目チェック必須。通過履歴の記録。「1冊を通してチェックする」関所機能 |
| **xlsx知見** | SRS案件カルテに「校正・原稿整理チェックリスト（初稿のみ）」シートあり。チェックリストのパターンとして参考 |
| **実装優先度** | 高優先 |

#### FR-05: 進捗アラート

| 項目 | 内容 |
|------|------|
| **目的** | 顧客戻し遅れや工程滞留を早期検知。リマインド文面のAI生成 |
| **既存実装** | ❌ 未実装（ダッシュボードに遅延数表示はあるがアラート通知なし） |
| **新規内容** | 遅延タスク自動検出＋アラートバッジ。今週期限タスク一覧。顧客戻し待ちの経過日数表示。リマインド文面テンプレート生成（コピペ用） |
| **xlsx知見** | ダイハツ管理シートの「ダイハツ様確認中」列が顧客戻し待ち管理の参考 |
| **実装優先度** | 高優先 |

#### FR-06: スケジュール自動展開

| 項目 | 内容 |
|------|------|
| **目的** | 案件ランク（A〜D）に基づき、数ヶ月先までのスケジュールを一括自動生成 |
| **既存実装** | △ FMTシステムはあるがランク概念なし。手動で日付設定が必要 |
| **改修内容** | Projectにランク属性（A/B/C/D）追加。ランクごとにFMT工程の標準日数を定義。ランク＋開始日指定→日程自動算出。入稿物/取材物等のプロセスパターン別対応 |
| **xlsx知見** | ダイハツの「スケジュールテンプレ」シートが参考。ページ番号＋制作方法別の列テンプレート構造 |
| **実装優先度** | 中優先 |

#### FR-07: フォルダ自動生成

| 項目 | 内容 |
|------|------|
| **目的** | 台割りからフォルダ構造を自動生成し、デスク作業を半減 |
| **既存実装** | △ フォルダDB機能あり（命名規則設定・ツリープレビュー）。実ファイルシステム操作は非対応 |
| **改修内容** | 既存フォルダDB機能を拡張。台割データからフォルダ構造の自動提案。生成されたフォルダ構造のテキスト出力（手動作成ガイド） |
| **備考** | 議事録にも「要検証」と記載あり。ブラウザのFile System Access APIの制約内で実装 |
| **実装優先度** | 中優先 |

#### FR-08: データインポート

| 項目 | 内容 |
|------|------|
| **目的** | 瀬山さん運用中のスプレッドシートのデータを読み込み、初期構築のベースとする |
| **既存実装** | ✅ CSVインポート実装済み（エンコーディング自動検出、カラムマッピングUI） |
| **改修内容** | 既存CSV機能で基本対応可能。必要に応じてxlsx直接読み込み検討（外部ライブラリ要） |
| **実装優先度** | 最優先 |

### 2-2. xlsx分析から追加する管理項目

bushi5のxlsx分析（75シート精読）から、議事録FR-01〜08に含まれていない運用上必須の管理項目を補完する。

#### FR-09: 記事/コンテンツ管理（情報とりためシート機能）

| 項目 | 内容 |
|------|------|
| **目的** | 記事単位の詳細情報（内容・文字量・ソース元・要望・資料リンク・備忘録）を一元管理 |
| **既存実装** | ❌ 未実装（タスクにnoteフィールドのみ） |
| **新規内容** | タスクの詳細パネルを拡張: 記事名/記事内容概要/文字量/掲載スペース/ソース元（取材/資料/寄稿）/クライアント要望/資料リンク（URL）/備忘録・Q&A記録/確認中事項 |
| **xlsx出典** | ダイハツ管理シート「情報とりためシート」（制作管理のハブ。全3ファイル中最重要シート） |
| **実装優先度** | 高優先 |

#### FR-10: 台割（ページ構成）管理

| 項目 | 内容 |
|------|------|
| **目的** | 号のページ構成を見開き単位で管理。制作方法（取材/資料/寄稿）と担当者をマッピング |
| **既存実装** | ❌ 未実装 |
| **新規内容** | 台割ビュー: ページ番号/記事タイトル/制作方法/担当者マッピング/開き方向/ステータス |
| **xlsx出典** | ダイハツ「台割シート」（最新/提案用/old 計5シート） |
| **実装優先度** | 中優先 |

#### FR-11: 年間スケジュール俯瞰

| 項目 | 内容 |
|------|------|
| **目的** | 複数号（6号分等）を年間カレンダーで俯瞰し、全体のリソース計画に活用 |
| **既存実装** | ❌ 未実装 |
| **新規内容** | 年間カレンダービュー: 月別の号/案件の配置。主要マイルストーン表示。ガントチャートとの階層的ドリルダウン |
| **xlsx出典** | ダイハツ「年間スケジュール」（5列1セット×6号分の俯瞰構造） |
| **実装優先度** | 低優先 |

#### FR-12: 案件基本情報・顧客管理

| 項目 | 内容 |
|------|------|
| **目的** | 案件ごとの基本情報（クライアント/媒体/仕様/JBA体制/取引経緯）を一元管理。属人知識の形式知化 |
| **既存実装** | △ プロジェクト名のみ |
| **新規内容** | Projectに拡張属性: クライアント名/媒体名/号数/仕様（版型・ページ数）/JBA担当体制/クライアント担当者/取引年数/発行目的/ターゲット |
| **xlsx出典** | SRS案件カルテ「基本情報」（270行×27列の包括ドキュメント。担当者の性格・注意点まで記録） |
| **実装優先度** | 低優先 |

## 3. 非機能要件

| ID | 内容 | 出典 | 対応方針 |
|----|------|------|---------|
| NFR-01 | Webアプリケーション形式（HTML+CSS+JS）でブラウザ動作 | 議事録 | 既存プロトタイプ踏襲 |
| NFR-02 | 運用コストは無料 | 文字起こしL20 | サーバーレス（ローカルHTML）で対応 |
| NFR-03 | ビジュアル面でZoho等より圧倒的に優れていること | 文字起こしL21 | ダークテーマの統一感を維持・強化 |
| NFR-04 | 外部サービス連携は当面なし。リマインドはAI生成→手動コピペ | 議事録L39 | Slack等のAPI連携は不要 |
| NFR-05 | 大阪50案件規模を横並び一覧表示可能 | 文字起こしL7,L10 | パフォーマンス最適化必須 |
| NFR-06 | 日付バリデーション（終了日 < 開始日の防止） | bushi4分析 | 入力時バリデーション追加 |
| NFR-07 | データバックアップ/復元機能 | bushi4分析 | localStorage + JSON export/import |
| NFR-08 | undo/redo機能 | bushi4分析 | 操作履歴スタック実装 |

## 5. 画面構成・ビュー定義

### 5-1. 既存ビュー（改修）

#### ビュー1: ダッシュボード（プロジェクト一覧）

| 項目 | 仕様 |
|------|------|
| 表示内容 | プロジェクトカード一覧。各カードに進捗バー + タスク数/完了/遅延/担当者 |
| 改修内容 | アラートバッジ追加（遅延タスク数）。今週期限のタスク一覧パネル追加。全プロジェクト横断の進捗サマリー強化 |
| 操作 | プロジェクト追加/リネーム/削除/クリックで詳細画面へ |

#### ビュー2: テーブルビュー（タスク一覧）

| 項目 | 仕様 |
|------|------|
| 表示内容 | タスクの全フィールドをグリッド表示 |
| 改修内容 | 担当者カラム追加。ステータスフィルタ/担当者フィルタ/検索ボックス追加。品質関所マーク表示 |
| 操作 | インライン編集/ドラッグ＆ドロップ並び替え/担当者変更（ボール移動） |

#### ビュー3: ガントチャート

| 項目 | 仕様 |
|------|------|
| 表示内容 | 月別カレンダー×タスク/ページ別のマトリクス |
| 改修内容 | 担当者ラベル追加。横スクロール改善（50案件対応）。工程間の依存関係表示（矢印） |
| 操作 | 日付範囲選択/バーのドラッグで日程変更/今日線クリックで今日にスクロール |

## 7. Phase3 サブタスク分割案

### 7-1. 分割方針

既存HTMLの1344行をベースに、機能要件の実装優先度に従い4フェーズで分割する。各フェーズを1〜2柱に割り当て、ファイル競合（RACE-001）を回避する。

**重要**: 単一HTMLファイル構成のため、同時編集は不可。フェーズ内の並列化はファイル分離（モジュール化）が前提。

### 7-2. 推奨アーキテクチャ変更

既存の1344行モノリシックHTMLを、Phase3開始前に以下のモジュール構成に分離することを推奨する:

```
output/スケジュール/
├── index.html          ← メインHTML（構造のみ）
├── css/
│   └── style.css       ← スタイルシート
└── js/
    ├── app.js           ← 初期化・状態管理・ルーティング
    ├── data.js          ← データ構造定義・localStorage I/O・マイグレーション
    ├── members.js       ← 担当者マスター管理
    ├── dashboard.js     ← ダッシュボード + アラート
    ├── table.js         ← テーブルビュー
    ├── gantt.js         ← ガントチャート
    ├── heatmap.js       ← 稼働ヒートマップ
    ├── quality.js       ← 品質関所
    ├── daiwari.js       ← 台割ビュー
    ├── fmt.js           ← FMT + ランクテンプレート
    ├── csv.js           ← CSV入出力
    └── folder.js        ← フォルダDB
```

この分離により:
- 各柱が異なるJSモジュールを担当でき、ファイル競合を回避
- コード品質が向上（bushi4指摘の「モノリシック構造」解消）
- テスト容易性も向上
